# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

The Quartz static site that publishes https://wiki.theloop.community. Its
`gh-pages` branch is what GitHub Pages actually serves; `main` holds the
site config, theme, and plugin sources.

This repo (`djsprenk/theloopwiki`) and the separate
`theloopwiki-tools` repo are **two separate git repos with two separate
GitHub remotes** — see the parent `theloopwiki-tools/CLAUDE.md` for the
full rationale. Content is filtered and pushed into `../content` by that
repo's `publish.sh` / `filter.py`; this repo never touches the vault
directly.

## Commands

Local dev/preview (serves whatever is in `../content`, no deploy):

```
npx quartz build --serve --directory ../content
```

Quartz's own test/lint commands:

```
npm run check   # tsc --noEmit && prettier --check
npm test        # tsx --test
```

## Locally-forked plugins

`.quartz/plugins/` is gitignored and managed by Quartz's plugin CLI
(`npx quartz plugin add/install/update`) — anything edited there directly
is lost the next time plugins are reinstalled or synced from
`quartz.config.yaml`/`quartz.lock.json`.

When a plugin's exposed config options aren't enough (e.g. changing literal
text/markup, not just data), fork it into `local-plugins/<name>/` instead,
which **is** tracked in git. Register it with:

```
npx quartz plugin add ./local-plugins/<name>
```

This symlinks `local-plugins/<name>` into `.quartz/plugins/<name>` and
updates `source:` in `quartz.config.yaml` to the local path — the fork
survives future `.quartz/` regeneration.

**Gotcha:** after editing a local fork's `src/`, `npx quartz plugin update
<name>` (or `install --latest`) does NOT recompile it — it reports "using
pre-built dist/" and silently keeps serving the stale build. Rebuild
manually:

```
cd local-plugins/<name>
npx tsup
```

Then re-run `npx quartz build` to pick up the new `dist/`.

Current forks:

- `footer` (`local-plugins/footer/`) — the upstream
  `github:quartz-community/footer` plugin only exposes a `links` option,
  not the footer message text. Forked to render "The Loop © {year} -
  Created with Quartz v{version}", with "The Loop" linking to
  https://theloop.community. The `GitHub`/`Discord Community` links list
  was removed by setting `options.links: {}` on this plugin's entry in
  `quartz.config.yaml`, rather than in the fork itself.
- `folder-page` (`local-plugins/folder-page/`) — the upstream
  `github:quartz-community/folder-page` plugin's `FolderContent` component
  always appends the auto-generated by-date file listing below any content
  in a folder's index note, with no option to suppress it. Forked to add
  `hideListingWhenIndexed` (default `true`): when the folder's index page
  has real authored content — either a literal `index.md`, or the Obsidian
  "folder note" convention (a file named identically to its parent folder,
  e.g. `The Loop Academy/The Loop Academy.md`) — the listing is skipped
  entirely and only the authored content shows. Folders with no authored
  index still get the normal listing (this is how "has real content" is
  detected: the rendered HAST tree is non-empty vs. empty for
  auto-generated virtual folder pages).

## Folder click → specific page (not the auto listing)

Two related but distinct customizations exist for folder index pages:

- **Suppress the listing, keep your own content**: write real content in
  the folder's index note (`index.md`, or the folder-note-with-matching-name
  convention above). The `folder-page` fork's `hideListingWhenIndexed`
  handles the rest automatically — no extra frontmatter needed.
- **Redirect straight to a different page entirely** (skip even your own
  index content): add `aliases:` to the *target* page's frontmatter, with
  the full absolute slug of the folder's index page, e.g.:

  ```yaml
  aliases:
    - the-loop-academy/01-foundations-of-djing/index
  ```

  Must be the **full absolute slug**, not a relative path — `alias-redirects`'s
  relative-URL resolver explicitly refuses anything ending in `index`, so a
  bare `index` alias resolves to the site root instead of the folder. This
  works reliably (not a race) because Quartz's build (`quartz/build.ts`)
  always runs `PageTypeDispatcher` — which renders the folder's own
  listing/content page — in Phase 1, then all other emitters, including
  `AliasRedirects`, in Phase 2. Phase 2's redirect HTML always overwrites
  the Phase 1 output at that path.

## Encrypted (password-protected) content

Course content is published with per-page passwords via the
`encrypted-pages` community plugin (AES-256-GCM, decrypted client-side).
Every note under `The Loop Academy/` currently has a `password:` frontmatter
field, which means:

- Built HTML for those pages has no visible content in the raw output —
  the entire HAST tree is replaced with an opaque ciphertext container at
  build time. `grep`-ing built HTML/CSS for page content will find nothing;
  verifying visual/content changes requires an actual browser with the
  password entered.
- `static/encryptedContentIndex.json` is a companion "shadow" content index
  for `unlisted: true` encrypted pages, regenerated on every build.

See `docs/plugins/EncryptedPages.md` for the full plugin behavior
(`unlisted` vs `stealth` pages, shadow index format, config options).
